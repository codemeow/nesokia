# Макрогенерация специализированного кода размещения объектов для ассемблера ca65

## Аннотация

При разработке программ для процессора MOS 6502 — платформы с жёсткими
ограничениями по ресурсам, широко применяемой в ретро-разработке (NES, Atari и
совместимые системы) — критически важна эффективность операций добавления
объектов в пулы. Существующие подходы вынуждают разработчика выбирать между
удобством единого интерфейса и скоростью выполнения. В настоящей работе
предлагается макрос для ассемблера ca65, выполняющий статический анализ режимов
адресации аргументов, автоматически формирующий план переключений регистров и
генерирующий специализированный код без накладных расходов времени выполнения.
Предложенный метод сокращает время выполнения операции размещения до ~70% от
времени выполнения реализации через временные переменные в типичном случае и до
~50% при использовании только непосредственной адресации, сохраняя при этом
единый удобный интерфейс вызова.

---

## 1. Введение

Процессор MOS 6502 по-прежнему актуален в области ретро-разработки и служит
удобной моделью для изучения оптимизации кода в условиях жёстких ресурсных
ограничений. Стоимость каждой инструкции и каждого такта исполнения критически
важна в коде, выполняемом внутри игровых циклов или обработчиков графических
данных.

Одной из распространённых задач является добавление объектов в линейные
структуры данных — буферы спрайтов (OAM) или пулы объектов. Наивная реализация
через универсальные подпрограммы или временные переменные упрощает интерфейс,
но приводит к постоянным накладным расходам. Ручная специализация кода даёт
максимальную скорость, однако плохо масштабируется: сопровождение становится
сложным при большом числе объектов и вариантов их размещения.

В настоящей работе предлагается механизм генерации специализированного кода с
помощью макросов ассемблера ca65. В разделе 2 формулируется задача и описываются
ограничения существующих подходов. В разделе 3 излагается предлагаемый алгоритм.
Раздел 4 содержит пример сгенерированного кода и оценку производительности.

---

## 2. Постановка задачи

### 2.1. Общая схема размещения объектов

Рассмотрим задачу добавления объектов в пул. Пусть пул хранится в виде
нескольких параллельных массивов, каждый из которых содержит одно свойство
объектов. Обозначим через `pool_size` текущее число объектов в пуле, через
`POOL_MAX` — максимально допустимое число объектов. Свойства объекта хранятся
в массивах `pool_object`, `pool_flags`, `pool_timer_a`. Добавление нового
объекта требует записи значений во все массивы по индексу `pool_size`, после
чего `pool_size` должно быть увеличено на единицу.

### 2.2. Простейшая реализация

Рассмотрим реализацию добавления объекта с константными значениями свойств:

```asm
    ldx pool_size
    cpx #POOL_MAX
    beq :+

    lda #$05
    sta pool_object, x
    lda #$19
    sta pool_flags, x
    lda #$98
    sta pool_timer_a, x
    inc pool_size
:
```

Такой код работает корректно, однако на практике значения свойств часто
вычисляются или извлекаются из других структур данных.

### 2.3. Использование внешних массивов данных

Рассмотрим ситуацию, когда часть данных объекта хранится в отдельном массиве —
например, список тайлов спрайта хранится в таблице кадров анимации. Регистр X
содержит смещение в OAM-буфере (`sprites_used`), регистр Y — индекс текущего
кадра:

```asm
sprites:
    .byte $80, $81, $82
...
ldx sprites_used    ; X = позиция в OAM-буфере
ldy #$02            ; Y = индекс спрайта в кадре
lda sprites, y
sta OAMBUFFER + 1, x
inx                 ; переход к следующей позиции в OAM
```

Здесь X и Y несут разные роли, что само по себе корректно. Сложность возникает,
когда требуется одновременно работать с несколькими массивами, индексируемыми
разными регистрами.

### 2.4. Конфликты регистров

Предположим, в регистре Y хранится индекс кадра анимации, а в регистре X — номер
мира, определяющий атрибуты спрайта. Чтобы обратиться к обоим массивам,
необходимо временно сохранять регистры:

```asm
sprites:
    .byte $80, $81, $82
attrs:
    .byte $02, $01, $02
...

; читаем тайл по индексу кадра (Y)
txa
pha                         ; сохраняем X (номер мира)
ldx sprites_used
lda sprites, y
sta OAMBUFFER + 1, x
pla
tax

; читаем атрибут по номеру мира (X)
tya
pha                         ; сохраняем Y (индекс кадра)
ldy sprites_used
lda attrs, x
sta OAMBUFFER + 2, y
pla
tay
```

Необходимость сохранять и восстанавливать регистры увеличивает число инструкций
и усложняет код.

### 2.5. Использование промежуточных переменных

Альтернативный подход — заранее скопировать нужные значения во временные
переменные, а затем работать с ними через единственный индексный регистр:

```asm
lda sprites, y
sta _sprite_index       ; сохраняем тайл во временную переменную

lda attrs, x
sta _sprite_attr        ; сохраняем атрибут

txa
pha                     ; сохраняем X

ldx sprites_used

lda _sprite_index
sta OAMBUFFER + 1, x

lda _sprite_attr
sta OAMBUFFER + 2, x

pla
tax
```

Такой подход упрощает управление регистрами, однако требует дополнительных
переменных (желательно в Zero Page) и выполняет лишние операции сохранения даже
тогда, когда конфликта регистров нет.

### 2.6. Ограничения на автоматизацию

Если попытаться оформить размещение объекта как макрос принимающий несколько
аргументов, возникает проблема: универсальная реализация вынуждена всегда
прибегать к временным переменным или нескольким версиям макроса для каждой
комбинации режимов адресации. Это плохо масштабируется — при десяти и более
свойствах объекта количество таких версий становится неприемлемым.

В результате разработчик оказывается перед выбором между тремя неудобными
вариантами:

* использовать универсальный, но неоптимальный макрос;
* создавать большое количество специализированных макросов для различных
комбинаций регистров;
* каждый раз писать код размещения вручную.

Требуется механизм, обеспечивающий одновременно: единый интерфейс вызова,
поддержку произвольного числа свойств и отсутствие накладных расходов времени
выполнения.

## 3. Описание решения

### 3.1. Общая идея алгоритма

Для реализации оптимизированного макроса предлагается использовать следующий
интерфейс:

```asm
nsk_pool_add prop1, prop2, prop3 ...
```

Основная задача макроса — сгенерировать последовательность инструкций, которая:

* корректно обрабатывает различные режимы адресации аргументов;
* минимизирует число операций сохранения и восстановления регистров;
* генерирует код, близкий по эффективности к вручную написанному.

Для достижения этой цели используется следующий алгоритм:

1. Анализ аргументов макроса — определение режима адресации каждого аргумента.
2. Формирование плана переключений — построение минимальной последовательности
смен регистра.
3. Сохранение регистров в стек в обратном порядке согласно плану (поскольку стек
имеет структуру FILO).
4. Генерация кода записи — регистры извлекаются из стека только в момент смены
режима адресации.

При генерации кода для адресации целевых массивов пула используется регистр
**X**, а для адресации пользовательских данных — регистр **Y**. Если
пользователь передаёт аргумент с адресацией `,x`, макрос переключает его на
`,y`, используя тот факт, что `ptr,x` и `ptr,y` дают одинаковый результат при
равенстве значений регистров.

### 3.2. Анализ режимов адресации

На первом этапе для каждого аргумента определяется режим адресации.
Поддерживаются три режима - immediate, X indexed и Y indexed; косвенная
адресация `(ptr),y` и `(ptr,x)` в рамках настоящего решения не рассматривается:

```asm
.define _PA_ARG_TYPE_NONINDEXED 0 ; ptr / #value — без индексного регистра
.define _PA_ARG_TYPE_ABSINDX    1 ; ptr, x
.define _PA_ARG_TYPE_ABSINDY    2 ; ptr, y

.define _NSK_VAR_ARGTYPE_(index) .ident(.sprintf("argtype_%d", index))
```

Результат для каждого аргумента сохраняется в виртуальной переменной
`argtype_N`. Дополнительно ведётся флаг `_xclobbered`: он устанавливается в 1,
если среди аргументов текущего вызова хотя бы один использует адресацию `,x`.
Если флаг равен 0, сохранять регистр **Y** в стек не требуется.

Макрос анализа одного аргумента:

```asm
.macro _PA_ARG_CHECK arg

    .if (.xmatch (.left (1, {arg}), #))
        _NSK_VAR_ARGTYPE_ { _argindex } .set _PA_ARG_TYPE_NONINDEXED

    .elseif (.xmatch (.right (1, {arg}), x))
        _NSK_VAR_ARGTYPE_ { _argindex } .set _PA_ARG_TYPE_ABSINDX
        _xclobbered .set 1

    .elseif (.xmatch (.right (1, {arg}), y))
        _NSK_VAR_ARGTYPE_ { _argindex } .set _PA_ARG_TYPE_ABSINDY

    .else
        _NSK_VAR_ARGTYPE_ { _argindex } .set _PA_ARG_TYPE_NONINDEXED
    .endif

    _argindex .set _argindex + 1

.endmacro
```

Пример использования внутри основного макроса:

```asm
.macro nsk_pool_add prop1, prop2, prop3, ...

    _argindex   .set 0
    _xclobbered .set 0

    _PA_ARG_CHECK { prop1 }
    _PA_ARG_CHECK { prop2 }
    _PA_ARG_CHECK { prop3 }
    ; ... остальные свойства

    ...
.endmacro
```

### 3.3. Пример анализа аргументов

Рассмотрим вызов макроса с восемью аргументами:

```asm
nsk_pool_add \
    { ptr1, x }, \
    { ptr2, y }, \
    { ptr3, x }, \
    { #val1   }, \
    { ptr4, x }, \
    { ptr5, x }, \
    { #val2   }, \
    { ptr6, y }
```

После анализа будут сформированы следующие переменные:

```
argtype_0 = _PA_ARG_TYPE_ABSINDX
argtype_1 = _PA_ARG_TYPE_ABSINDY
argtype_2 = _PA_ARG_TYPE_ABSINDX
argtype_3 = _PA_ARG_TYPE_NONINDEXED
argtype_4 = _PA_ARG_TYPE_ABSINDX
argtype_5 = _PA_ARG_TYPE_ABSINDX
argtype_6 = _PA_ARG_TYPE_NONINDEXED
argtype_7 = _PA_ARG_TYPE_ABSINDY
```

### 3.4. Формирование плана переключений регистров

На следующем этапе строится минимальная последовательность смен режима
адресации. Аргументы типа `NONINDEXED` пропускаются, так как не требуют
индексного регистра. Каждый раз, когда режим меняется с `ABSINDX` на `ABSINDY`
или обратно, фиксируется точка переключения.

```asm
.define _NSK_VAR_SWITCH_(index) .ident(.sprintf("switch_%d", index))
```

Макросы сохранения регистров:

```asm
.macro _PA_X_PUSH
    txa
    pha
.endmacro

.macro _PA_Y_PUSH
    .if _xclobbered = 1     ; сохраняем Y только если X используется в аргументах
        tya
        pha
    .endif
.endmacro
```

Макрос сохранения регистра в соответствии с типом аргумента:

```asm
.macro _PA_INDEX_PUSH argtype
    .if argtype = _PA_ARG_TYPE_ABSINDX
        txa
        pha
    .elseif argtype = _PA_ARG_TYPE_ABSINDY
        tya
        pha
    .endif
.endmacro
```

Построение списка точек переключения:

```asm
.macro _PA_SWITCH_ADD count, index

    .if (_NSK_VAR_ARGTYPE_ { index } <> _PA_ARG_TYPE_NONINDEXED) && \
        (_NSK_VAR_ARGTYPE_ { index } <> _currenttype)

        .repeat count, i
            .if _switchcount = i
                _NSK_VAR_SWITCH_ {i} .set _NSK_VAR_ARGTYPE_ { index }
            .endif
        .endrep

        _switchcount .set _switchcount + 1
        _currenttype .set _NSK_VAR_ARGTYPE_ { index }

    .endif

.endmacro
```

Построение полного плана и помещение регистров в стек:

```asm
.macro _PA_PLAN_BUILD count

    .repeat count, i
        _NSK_VAR_SWITCH_ { i } .set _PA_ARG_TYPE_NONINDEXED
    .endrep

    _switchcount .set 0
    _currenttype .set _PA_ARG_TYPE_ABSINDY  ; начальное значение: Y,
    ; Это позволяет пропустить tya, pha, pla, tay для первого элемента,
    ; если он индексирован по Y

    .repeat count, i
        _PA_SWITCH_ADD count, i
    .endrep

    .repeat _switchcount, i
        _checkindex .set _switchcount - 1 - i
        _PA_INDEX_PUSH _NSK_VAR_SWITCH_ { _checkindex }
    .endrep

.endmacro
```

Использование внутри основного макроса:

```asm
.macro nsk_pool_add prop1, prop2, prop3, ...
    ...

    _PA_X_PUSH
    _PA_Y_PUSH

    _PA_PLAN_BUILD .paramcount

    ...
.endmacro
```

Если среди аргументов нет ни одного с адресацией `,x` (флаг `_xclobbered = 0`),
сохранение Y пропускается — это экономит 11 тактов на последовательности
`tya/pha/pla/tay`.

### 3.5. Пример построения плана переключений

Для аргументов из раздела 3.3 с начальным значением `_currenttype = ABSINDY`:

| Аргумент  | Тип        | Смена режима?            |
|-----------|------------|--------------------------|
| argtype_0 | ABSINDX    | да (ABSINDY - ABSINDX)   |
| argtype_1 | ABSINDY    | да (ABSINDX - ABSINDY)   |
| argtype_2 | ABSINDX    | да (ABSINDY - ABSINDX)   |
| argtype_3 | NONINDEXED | пропуск                  |
| argtype_4 | ABSINDX    | нет                      |
| argtype_5 | ABSINDX    | нет                      |
| argtype_6 | NONINDEXED | пропуск                  |
| argtype_7 | ABSINDY    | да (ABSINDX - ABSINDY)   |

Формируются четыре точки переключения:

```
switch_0 = ABSINDX
switch_1 = ABSINDY
switch_2 = ABSINDX
switch_3 = ABSINDY
```

Поскольку стек имеет структуру FILO, регистры помещаются в стек в обратном
порядке:

```asm
tya    ; switch_3 = ABSINDY
pha
txa    ; switch_2 = ABSINDX
pha
tya    ; switch_1 = ABSINDY
pha
txa    ; switch_0 = ABSINDX
pha
```

### 3.6. Генерация кода записи

На заключительном этапе выполняется повторный проход по аргументам. В момент
смены режима адресации из стека извлекается значение соответствующего регистра
и загружается в Y. Регистр X при этом всегда содержит `pool_size` и используется
исключительно для записи в массивы пула.

Замечание: для аргументов с исходной адресацией `,x` (тип ABSINDX) чтение данных
также выполняется через Y (`lda ..., y`), а не через X. Это корректно, поскольку
в момент чтения Y содержит то же значение, которое пользователь передал в X (см.
обоснование в разделе 3.1).

```asm
.define _NSK_POOL_IDENT(name) .ident(.sprintf("nsk_pool_%s", name))
```

Макросы восстановления регистров:

```asm
.macro _PA_X_PULL
    pla
    tax
.endmacro

.macro _PA_Y_PULL
    .if _xclobbered = 1
        pla
        tay
    .endif
.endmacro
```

Макрос записи одного свойства:

```asm
.macro _PA_ARG_WRITE name, arg

    .if _NSK_VAR_ARGTYPE_ { _argindex } = _PA_ARG_TYPE_NONINDEXED
        lda arg

    .else

        .if _NSK_VAR_ARGTYPE_ { _argindex } <> _currenttype
            pla
            tay
            _currenttype .set _NSK_VAR_ARGTYPE_ { _argindex }
        .endif

        lda .left (.tcount ({arg}) - 2, {arg}), y

    .endif

    sta _NSK_POOL_IDENT { name }, x

    _argindex .set _argindex + 1

.endmacro
```

Основной макрос:

```asm
.macro nsk_pool_add prop1, prop2, prop3, ...
    ...

    _argindex    .set 0
    _currenttype .set _PA_ARG_TYPE_ABSINDY

    ldx nsk_pool_size

    _PA_ARG_WRITE "object",  { prop1 }
    _PA_ARG_WRITE "flags",   { prop2 }
    _PA_ARG_WRITE "timer_a", { prop3 }
    ; ... остальные свойства

    inc nsk_pool_size

    _PA_Y_PULL
    _PA_X_PULL

    ...
.endmacro
```

Таким образом, макрос автоматически генерирует последовательность инструкций,
минимизирующую число операций сохранения регистров и обеспечивающую эффективную
работу с произвольными комбинациями режимов адресации.

## 4. Результаты

### 4.1. Пример генерации кода

Рассмотрим вызов макроса `nsk_pool_add` с восемью явными аргументами для пула
из двенадцати свойств. Четыре свойства типа `_frac` (дробные части координат)
заполняются автоматически значением `#$00`, поскольку в данном вызове они не
передаются явно. Конкретные адреса приведены исключительно для наглядности:

```asm
nsk_pool_object       = $1000
nsk_pool_flags        = $1001
nsk_pool_worldx_hi    = $1010
nsk_pool_worldx_lo    = $1011
nsk_pool_worldx_frac  = $1012   ; заполняется автоматически (#$00)
nsk_pool_worldy_lo    = $1020
nsk_pool_worldy_frac  = $1021   ; заполняется автоматически (#$00)
nsk_pool_vectorx_lo   = $1030
nsk_pool_vectorx_frac = $1031   ; заполняется автоматически (#$00)
nsk_pool_vectory_lo   = $1040
nsk_pool_vectory_frac = $1041   ; заполняется автоматически (#$00)
nsk_pool_timer_a      = $1050

nsk_pool_size         = $2000
```

Вызов макроса (8 явных аргументов = 12 инструкций записи):

```asm
nsk_pool_add \
    { $1000  }, \   ; object   — NONINDEXED
    { $01, x }, \   ; flags    — ABSINDX
    { $10, y }, \   ; worldx_hi — ABSINDY
    { $11, y }, \   ; worldx_lo — ABSINDY
    { $20    }, \   ; worldy_lo — NONINDEXED
    { $30    }, \   ; vectorx_lo — NONINDEXED
    { $40, y }, \   ; vectory_lo — ABSINDY
    { $50, x }      ; timer_a  — ABSINDX
```

Сгенерированный машинный код:

```asm
; Проверка переполнения пула
ad 00 20   lda $2000
c9 40      cmp #$40
f0 5c      beq :+

; Сохранение пользовательских значений X и Y
8a         txa
48         pha
98         tya
48         pha

; Помещение плана переключений в стек (порядок: ABSINDX, ABSINDY, ABSINDX)
8a         txa
48         pha
98         tya
48         pha
8a         txa
48         pha

; X = текущий индекс пула
ae 00 20   ldx $2000
```

### 4.2. Оценка производительности

В таблице ниже приведено сравнение времени выполнения операции добавления
объекта для трёх реализаций на примере из раздела 4.1.

| Сценарий                               | Предложенный макрос  | Временные переменные (Zero Page)  | Временные переменные (вне Zero Page)  |
|----------------------------------------|----------------------|-----------------------------------|---------------------------------------|
| Смешанная адресация (пример из §4.1)   | **171–176 тактов**   | 192–197 тактов                    | 208–213 тактов                        |
| Все аргументы константные (NONINDEXED) | **113 тактов**       | 192–197 тактов                    | 208–213 тактов                        |

Реализация через временные переменные имеет фиксированную стоимость независимо
от режимов адресации: лишние операции сохранения выполняются даже тогда, когда
данные могут быть использованы напосредственно. Предложенный макрос адаптируется
к конкретному вызову: в оптимальном случае (все аргументы — константы) выигрыш
составляет около двух раз.

Разброс 171–176 тактов в типичном случае обусловлен зависимостью числа тактов
инструкций `lda` и `sta` от расположения операндов относительно границ страниц
памяти (page crossing penalty).

## 5. Заключение

В настоящей работе предложен механизм генерации специализированного кода
размещения объектов в пуле с использованием макросов ассемблера ca65. Ключевым
результатом является сокращение времени выполнения операции добавления объекта
с 192–213 до 171–176 тактов в типичном случае и до 113 тактов при использовании
только непосредственной адресации — при сохранении единого интерфейса вызова.

В отличие от реализаций на основе временных переменных, предложенный механизм
выполняет статический анализ аргументов на этапе компиляции и генерирует
минимально необходимую последовательность сохранений и восстановлений регистров.
Это позволяет избежать постоянных накладных расходов времени выполнения,
характерных для универсальных реализаций.

Предложенный подход демонстрирует, что макросистема ca65 позволяет реализовывать
нетривиальные алгоритмы анализа и генерации кода непосредственно на этапе
компиляции, обеспечивая высокоуровневый интерфейс без ухудшения характеристик
производительности итогового машинного кода. Разработанный механизм применим не
только для пулов объектов, но и в других задачах, связанных с генерацией
специализированных последовательностей инструкций: при формировании OAM-буферов,
обработке структурированных данных или автоматизации повторяющихся шаблонов
ассемблерного кода.
